# Story 1.1: Integrate Rabin-Karp Orbit Scanner

## Status
Draft

---

## Story

**As a** compiler developer,
**I want** Rabin-Karp orbit scanner integrated for C/C++/CPP2 with TableGen middle tuple,
**so that** cppfort can parse multiple C-family languages deterministically and emit patterns for Sea of Nodes IR construction.

---

## Acceptance Criteria

1. **Rabin-Karp implementation** provides O(n) deterministic pattern matching across C/C++/CPP2 grammar
2. **Orbit masks** disambiguate confix operators (`{}`, `[]`, `<>`, `()`) based on context host (expression, lambda, attribute, template)
3. **TableGen middle tuple** stores pattern specifications between orbit scanner output and Sea of Nodes IR
4. **Evidence accumulation** uses pragmatic heuristics (no ML/probability) for grammar selection
5. **Integration** with existing `PatternMatcher` and `Machine` infrastructure from `src/stage0/machine.h`
6. **Simultaneous multi-grammar support** loads C, C++, and CPP2 patterns side-by-side with disambiguation when multiple grammars match
7. **Performance** achieves sub-100ms parsing for files up to 10KB with all three grammars active

## Dev Notes

### Architecture Context

**Source:** [divergence-son-simple-n-way.md](file:///Users/jim/work/cppfort/docs/architecture/divergence-son-simple-n-way.md)

Cppfort diverges from Simple compiler's single-target approach. The orbit scanner sits BEFORE Sea of Nodes:

```
Orbit Scanner (multi-grammar, Rabin-Karp based)
    ↓
TableGen Middle Tuple (pattern specs)
    ↓
Sea of Nodes IR (source representation)
    ↓
Pattern-Based Lowering (n-way conversion)
    ↓
{C, C++, CPP2, MLIR dialects}
```

**Key Architectural Points:**
- Orbit scanner handles **incomplete/invalid code** (for linting)
- Sea of Nodes requires **complete/valid programs** (for compilation)
- TableGen patterns bridge orbit output → SoN node types

### Rabin-Karp Algorithm Details

**Algorithm:** Hierarchical power-of-2 multi-resolution Rabin-Karp hashing

**Key Innovation:** Hash at multiple window sizes simultaneously (1, 2, 4, 8, 16, 32, 64, ...)
- Enables pattern matching at all resolution levels in a single pass
- Small patterns (single chars: `{`, `[`) detected at level 1
- Medium patterns (keywords: `class`, `template`) detected at levels 2-8
- Large patterns (signatures, expressions) detected at levels 16-64

**Hyper-parameter:** Maximum window size based on longest speculative pattern win
- Default: 64 bytes (can be tuned based on corpus analysis)
- **Tuning strategy:** Analyze corpus to find 99th percentile of successful pattern lengths
- Trade-off: Higher max = more resolution levels = more memory, but better pattern coverage
- **Longest-match principle:** Greedy matching favors specificity over generality

**Hash Function:** Polynomial rolling hash with prime base (31)

**Collision Handling:** Deterministic verification at each level (no false positives)

**Implementation:**
```cpp
class RabinKarp {
    // Hierarchical hash storage: _hashes[i] = hash at window size 2^i
    std::array<uint64_t, 7> _hashes;  // Levels 0-6: windows 1,2,4,8,16,32,64
    uint64_t _prime = 31;
    size_t _maxLevel;  // Determined by longest pattern (hyper-parameter)

public:
    // Initialize with maximum window size (power of 2)
    RabinKarp(size_t maxWindowSize = 64);

    // Compute all hierarchical hashes for initial window
    void initialize(const char* text, size_t len);

    // Rolling update: shift window by 1 char, update all levels
    void update(char remove, char add);

    // Get hash at specific power-of-2 level (0=1byte, 1=2bytes, 2=4bytes, ...)
    uint64_t hashAt(size_t level) const { return _hashes[level]; }

    // Find all patterns across all resolution levels
    std::vector<Match> findAll(const char* text, const Pattern* patterns, size_t count);
};
```

**Example Usage:**
```cpp
RabinKarp rk(64);  // Max window = 64 bytes (2^6)
rk.initialize(source, 64);

// Now we have 7 simultaneous hashes:
// Level 0: hash of 1 byte  (2^0)
// Level 1: hash of 2 bytes (2^1)
// Level 2: hash of 4 bytes (2^2)
// Level 3: hash of 8 bytes (2^3)
// Level 4: hash of 16 bytes (2^4)
// Level 5: hash of 32 bytes (2^5)
// Level 6: hash of 64 bytes (2^6)

// Single rolling update maintains all 7 hashes in O(1)
rk.update(oldChar, newChar);
```

**Benefits of Hierarchical Approach:**

1. **Single-pass multi-resolution matching:** Detect patterns of all sizes in one scan
2. **O(1) rolling update:** All hash levels maintained simultaneously with constant-time updates
3. **Memory efficient:** Fixed array size based on hyper-parameter (typically 7-10 levels)
4. **Speculative win detection:** Longest successful pattern at each position determines optimal level
5. **Adaptive tuning:** Hyper-parameter can be adjusted based on corpus analysis to maximize hits

**Complexity Analysis:**
- Initialization: O(k × log(maxWindowSize)) where k = number of patterns
- Per-character update: O(log(maxWindowSize)) = O(1) for fixed max (typically 6-7 levels)
- Pattern lookup: O(1) hash table lookup per level
- Total: O(n × log(maxWindowSize)) ≈ O(n) for scanning n characters

### Orbit Mask System

**Context Hosts** (determines which orbit is active):
- Expression context: `[` = array index (confidence: high)
- Lambda context: `[` = capture list (confidence: high)
- Declaration context: `[` = attribute (confidence: medium)
- Template context: `<` = template args (confidence: low due to `<` operator conflict)

**Orbit Types:**
```cpp
enum class OrbitType : uint8_t {
    OpenBrace,      // {
    CloseBrace,     // }
    OpenBracket,    // [
    CloseBracket,   // ]
    OpenAngle,      // <
    CloseAngle,     // >
    OpenParen,      // (
    CloseParen      // )
};
```


### Simultaneous Multi-Grammar Loading Strategy

**Key Architecture Decision:** Load all three grammars (C, C++, CPP2) **side-by-side** in the same orbit scanner instance.

**Why Simultaneous Loading:**
1. **Mixed-mode files:** Many real codebases have C/C++/CPP2 mixed in same file (extern "C", legacy code)
2. **Context-dependent disambiguation:** Same syntax can be valid in multiple languages - orbit context determines which
3. **Single-pass efficiency:** One scan detects patterns for all three grammars simultaneously
4. **Graceful degradation:** If one grammar interpretation fails, alternatives are already available

**Pattern Priority Resolution (when multiple grammars match):**
```cpp
enum class GrammarPriority : uint8_t {
    CPP2 = 3,  // Highest priority (most specific)
    CPP = 2,   // Medium priority
    C = 1      // Lowest priority (most general)
};
```

**Disambiguation Rules (in priority order):**
1. **Longest match wins:** If multiple patterns match, use the longest one (greedy/maximal munch)
2. **Grammar priority:** If multiple grammars tie for longest match, use highest priority (CPP2 > C++ > C)
3. **Orbit context validation:** Verify winner is valid in current orbit context
4. **Fallback:** If all interpretations fail validation, report ambiguity with all candidates

**Rationale:** Longest speculative win provides maximum specificity and matches natural human parsing intuition.

**Examples:**

```cpp
// Example 1: Longest match wins
// Source: int** ptr;
// Matches:
//   - C:    "int" (3 chars)
//   - C++:  "int*" (4 chars)
//   - C++:  "int**" (5 chars) ← WINNER (longest)
// Resolution: "int**" from C++ (longest match)

// Example 2: Grammar priority breaks tie
// Source: int x = 0;
// Matches (all same length):
//   - C:    "int x = 0" (9 chars)
//   - C++:  "int x = 0" (9 chars)
//   - CPP2: "int x = 0" (9 chars) ← WINNER (highest priority)
// Resolution: CPP2 interpretation (priority tie-break)

// Example 3: Unique pattern
// Source: main: () -> int = { }
// Matches:
//   - CPP2: "main: () -> int = { }" (full pattern) ← WINNER (only match)
// Resolution: CPP2 (unique match)

// Example 4: Context validation required
// Source: class { int x; }
// Matches:
//   - C++:  "class { ... }" ← WINNER after context check
//   - C:    rejected (invalid anonymous class in C)
// Resolution: C++ (C fails orbit context validation)
```
